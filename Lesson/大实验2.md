## （6）Spark 与 Hadoop MapReduce 的性能差异讨论

Spark 和 Hadoop MapReduce 均为大数据分布式计算框架，但基于架构设计、数据处理模式的差异，二者在性能表现上存在显著区别，以下结合实验场景详细分析：

## 一、核心架构差异导致的性能基础不同

### 1. 数据存储与计算模式

- Hadoop MapReduce：采用 “磁盘优先” 架构，计算过程中中间结果需频繁写入磁盘（Map 输出写入本地磁盘，Reduce 读取磁盘数据计算），磁盘 I/O 成为性能瓶颈。
- Spark：采用 “内存优先” 架构，核心数据结构 RDD（弹性分布式数据集）支持将中间结果缓存在内存中，后续迭代计算（如机器学习模型训练、多轮数据聚合）可直接复用内存数据，大幅减少磁盘 I/O 开销。

### 2. 任务调度机制

- Hadoop MapReduce：任务粒度较粗，Map 和 Reduce 阶段严格分离，任务启动、资源申请需频繁与 YARN 交互，延迟较高，适合长时运行的批量任务。
- Spark：支持细粒度任务调度，通过 DAG（有向无环图）优化任务执行流程，可将多个依赖任务合并为一个阶段（Stage）执行，减少任务启动和资源申请的开销，延迟更低。

## 二、实验场景下的性能表现对比

### 1. 批量数据处理（如实验二 PV/UV 统计）

- Hadoop MapReduce：处理一次性批量统计任务时，需经历 “Map → 磁盘写入 → Reduce → 磁盘输出” 全流程，对于 999 行数据规模，性能差异不明显，但数据量扩大至 TB 级后，磁盘 I/O 耗时会显著增加。
- Spark：即使处理小规模数据，也可通过内存缓存避免磁盘写入，任务执行速度比 MapReduce 快 2-5 倍；若需多次调整统计维度（如新增按评分分组统计），可直接复用内存中的 DataFrame，无需重复读取原始数据，效率优势更突出。

### 2. 迭代计算（如实验三机器学习训练）

- Hadoop MapReduce：机器学习模型训练需多轮迭代（如逻辑回归的梯度下降），每轮迭代均需重新读取数据并写入中间结果，磁盘 I/O 开销呈线性增长，处理大规模特征数据时性能极差。
- Spark：支持将训练数据、模型参数缓存在内存中，多轮迭代仅需在内存中更新参数，无需重复读写磁盘，迭代任务性能比 MapReduce 快 10-100 倍，是实验三分布式机器学习的核心优势。

### 3. 流式 / 交互式计算

- Hadoop MapReduce：不支持流式数据处理，交互式查询需通过 Hive 等上层组件转换，响应延迟通常在分钟级，无法满足实时分析需求。
- Spark：提供 Spark Streaming、Structured Streaming 组件，支持准实时流式处理，同时 Spark SQL 可实现交互式查询（响应延迟在秒级），若实验中需动态调整分析维度（如实时筛选高评分图书的 PV/UV），Spark 可快速响应。

## 三、性能优化空间差异

### 1. 资源利用率

- Hadoop MapReduce：Map 和 Reduce 阶段资源分配相互独立，易出现资源闲置（如 Map 阶段结束后 Reduce 阶段才开始占用资源），资源利用率通常在 50%-70%。
- Spark：支持动态资源分配，可根据任务执行进度调整 CPU、内存资源占用，同时支持任务并行执行（如实验二中 PV 统计与 UV 统计可并行处理），资源利用率可达 80%-90%。

### 2. 容错机制效率

- Hadoop MapReduce：容错依赖磁盘数据，任务失败后需重新读取原始数据从头执行，容错成本高。
- Spark：基于 RDD 的 lineage（血统）机制，任务失败后无需重新计算全量数据，仅需根据依赖关系重算丢失的分区，容错效率更高，尤其适合长时间运行的迭代任务。

## 四、总结：性能差异的核心结论

|对比维度|Hadoop MapReduce|Spark|
|---|---|---|
|核心优势|稳定性高、适合超大批量离线任务、硬件要求低|速度快、支持迭代 / 流式 / 交互式计算、资源利用率高|
|性能瓶颈|磁盘 I/O 频繁、迭代计算效率低|内存占用较高、小规模数据启动延迟略高|
|实验场景适配|仅适合一次性批量统计（如实验二基础 PV/UV 计算）|适配全场景（实验二统计分析、实验三机器学习训练）|

Spark 通过内存缓存、DAG 优化等设计，在绝大多数大数据分析场景中性能优于 Hadoop MapReduce，尤其适合需要多轮迭代、交互式查询的实验任务；而 Hadoop MapReduce 因稳定性高、硬件要求低，仍适用于超大规模离线批量处理（如 PB 级日志归档分析）。
